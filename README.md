고급 언어와 저급 언어
사람이 읽고 쓰기 쉬운 언어를 고급 언어라고 부른다. 예를 들어 C, Python, Java가 여기에 들어간다.
반대로 CPU가 더 직접 이해하는 형태에 가까운 언어를 저급 언어라고 부른다.
기계어: CPU가 직접 실행하는 0과 1
어셈블리어: 기계어를 사람이 읽기 쉽게 바꾼 기호 표현
고급 언어는 사람이 이해하기 쉽고, 저급 언어는 CPU에 가깝다. 둘은 경쟁 관계가 아니라 역할이 다른 표현 방식이다.

컴파일 언어와 인터프리터 언어
컴파일 언어는 보통 실행 전에 기계어에 가까운 형태로 번역된다.
예:
C
C++
Rust
인터프리터 언어는 실행 시점에 해석하며 동작하는 경우가 많다.
예: Python
JavaScript
다만 실제 구현은 더 복잡하다. 예를 들어 Python도 내부적으로 바이트코드 형태를 사용하고, Java는 바이트코드와 가상 머신을 사용한다. 초급 단계에서는 "실행 전에 번역되는 경향"과 "실행하면서 해석되는 경향" 정도로 이해해도 충분하다.

기계어와 어셈블리어를 왜 함께 보는가
학습자는 종종 "어셈블리어를 배우면 기계어를 다 아는 것인가?"라고 묻는다. 정확히는 아니다. 어셈블리어는 사람이 읽기 쉬운 기호 표현이고, 기계어는 CPU가 직접 실행하는 비트 패턴이다. 다만 두 표현은 같은 명령을 서로 다른 층에서 보여 준다는 점에서 강하게 연결되어 있다.
이 그림의 핵심은 "어셈블리어는 전혀 다른 언어"가 아니라, 기계어를 사람이 읽을 수 있게 표현한 중간 다리라는 점이다. 그래서 objdump 결과를 읽을 때도, 실제 CPU는 그 옆에 있는 비트 패턴이나 바이트열을 기준으로 움직인다고 이해하면 된다.

목적 파일 vs 실행 파일
소스 코드가 곧바로 최종 실행 파일이 되는 것은 아니다. 대표 흐름은 다음과 같다.
소스 파일(sample.c)
전처리 결과(sample.i)
어셈블리 파일(sample.s)
목적 파일(sample.o)
실행 파일(sample)
목적 파일은 아직 완전한 프로그램이 아니다. 여러 목적 파일과 라이브러리를 연결하는 링크(link) 과정을 거쳐 최종 실행 파일이 된다.
즉, 목적 파일은 중간 산출물, 실행 파일은 바로 실행 가능한 결과물에 가깝다.
좀 더 알아보기: 전처리, 컴파일, 어셈블, 링크
이 네 단계는 이름이 비슷해서 처음에는 쉽게 섞인다.
전처리
#include, #define 같은 전처리 지시문을 실제 코드로 펼친다.
컴파일
고급 언어를 어셈블리 수준으로 번역한다.
어셈블
어셈블리 코드를 목적 파일로 바꾼다.
링크
여러 목적 파일과 라이브러리를 묶어 실행 파일을 만든다.
이 과정을 이해하면 gcc -E, gcc -S, gcc -c 옵션이 왜 서로 다른 결과를 만드는지 이해하기 쉬워진다.

이 그림에서 가장 중요한 점은 사람이 쓰는 코드와 CPU가 실행하는 코드 사이에 여러 단계의 번역이 있다는 사실이다. 따라서 .o, .a, .so, 실행 파일은 모두 "어떤 단계의 산출물이냐"라는 관점에서 읽어야 한다.

컴파일 흐름을 읽을 때는 다음처럼 질문을 순서대로 던지면 도움이 된다.
 - 지금 보고 있는 파일은 사람이 읽는 텍스트인가
 - 이 파일은 아직 링크 전 단계인가
 - 실행 가능한 ELF 파일인가
 - 이 바이너리는 어떤 아키텍처를 대상으로 만들어졌는가

 - 저급 언어
 - 컴파일 언어
 - 인터프리터 언어
 - 목적 파일
 - 실행 파일

3-2 명령어의 구조

연산 코드와 오퍼랜드
명령어는 보통 다음 두 요소로 나눠 이해한다.
 - 연산 코드(opcode): 무엇을 할지
 - 오퍼랜드(operand): 무엇을 대상으로 할지
예를 들어 ADD R1, R2라는 형태를 생각해 보자.
 - ADD는 연산 코드
 - R1, R2는 오퍼랜드
즉, 명령어는 "무슨 일을 할지"와 "무엇에 대해 할지"의 조합이다.

명령어를 큰 부류로 나누면
실제 ISA는 훨씬 복잡하지만, 초급 단계에서는 명령어를 다음 정도로 나누어 이해하면 좋다.
 - 데이터 이동 명령
    MOV, PUSH, POP
 - 산술/논리 명령
   ADD, SUB, AND, OR
 - 비교/분기 명령
   CMP, JMP, JE, JNE
 - 함수 호출 제어 명령
   CALL, RET

이 분류는 단순 암기용이 아니라, 역어셈블 결과를 볼 때 "지금 이 코드는 값을 옮기는가, 계산하는가, 흐름을 바꾸는가"를 빠르게 읽게 해 준다.

4. CPU와 작동 원리
4-1 ALU와 제어장치

ALU : ALU(Arithmetic Logic Unit)는 산술 연산과 논리 연산을 담당한다.
 - 덧셈
 - 뺄셈
 - AND/OR/XOR
 - 시프트
 - 비교
초급 단계에서 중요한 점은 ALU가 "숫자를 계산하는 칸"이 아니라, CPU가 실제 연산을 수행하는 핵심 블록이라는 사실이다.

상태 플래그
ALU 연산은 종종 플래그 레지스터에도 영향을 준다. 예를 들어:
 - Zero flag: 결과가 0인가
 - Sign flag: 결과 부호는 어떤가
 - Carry flag: 자리올림이 생겼는가
 - Overflow flag: 부호 있는 연산 범위를 넘었는가
분기 명령은 이런 플래그를 보고 흐름을 바꾸기도 한다. 즉, 비교와 조건 분기는 ALU 결과와 밀접하게 연결된다.

제어장치
제어장치는 명령을 해석하고 CPU 내부 각 부분이 어떤 순서로 동작해야 하는지 지시한다.
제어장치가 맡는 일:
 - 명령어 인출 순서 조정
 - 명령 해독
 - 필요한 레지스터 선택
 - ALU 동작 지시
 - 메모리 읽기/쓰기 제어
즉, ALU가 "일꾼"이라면 제어장치는 "지휘자"에 가깝다.

4-2 레지스터
레지스터는 CPU 내부의 매우 빠른 저장 공간이다. 캐시보다도 더 CPU에 가깝다. 책 목차 기준으로 특히 다음은 꼭 알아 둘 필요가 있다.
  레지스터                        역할
프로그램 카운터(PC/RIP)         다음에 실행할 명령어 위치
명령어 레지스터(IR)             현재 해석 중인 명령어
스택 포인터(SP/RSP)             스택 꼭대기 위치
베이스 포인터(BP/RBP)           현재 함수 프레임의 기준 위치
범용 레지스터                   연산과 데이터 이동

반드시 알아야 할 레지스터
프로그램 카운터  -  다음 명령의 주소를 가리킨다.
명령어 레지스터  -  방금 인출한 명령을 담는다.
스택 포인터  -  스택의 현재 꼭대기 위치를 나타낸다.
베이스 포인터  -  현재 함수 프레임에서 기준점 역할을 한다.

특정 레지스터를 이용한 주소 지정 방식(1): 스택 주소 지정 방식
함수 호출 시 스택은 다음과 연결된다.
 - 반환 주소
 - 이전 프레임 정보
 - 지역 변수
 - 일부 매개변수
그래서 PUSH, POP, CALL, RET 같은 명령은 스택 포인터와 밀접하다. 함수 호출을 이해하려면 스택이 데이터 구조이면서 동시에 제어 흐름 구조라는 점을 알아야 한다.


특정 레지스터를 이용한 주소 지정 방식(2): 변위 주소 지정 방식
예를 들어 RBP-8 같은 표현은 "기준 레지스터에서 몇 바이트 떨어진 위치"를 뜻한다. 컴파일된 코드에서 지역 변수를 설명할 때 자주 등장한다.
이 방식이 중요한 이유는, 학습자가 어셈블리에서 [rbp-4], [rsp+16] 같은 표현을 봤을 때 "이상한 메모리 값"으로 느끼지 않고 "함수 프레임 안의 상대 위치"로 읽을 수 있게 해 주기 때문이다.

좀 더 알아보기: 상용화된 CPU 속 레지스터 및 주소 지정 방식
상용 CPU는 교재 그림보다 훨씬 많은 레지스터를 가진다. x86-64에는 rax, rbx, rcx, rdx, rsi, rdi, rsp, rbp, r8~r15 같은 레지스터가 있다. ARM64에도 별도의 범용 레지스터 체계가 있다.
중요한 점은 이름을 모두 외우는 것이 아니라 다음 질문에 답할 수 있는 것이다.
 - 다음 명령 위치를 누가 기억하는가
 - 함수 호출 시 현재 프레임 기준이 무엇인가
 - 데이터는 레지스터와 메모리 사이를 어떻게 오가는가

 - 스택 포인터
 - 베이스 포인터
 - 명령어 사이클
 - 인터럽트
 - 예외

 4-3 명령어 사이클과 인터럽트
명령어 사이클
CPU가 명령을 처리하는 기본 흐름은 보통 다음처럼 설명한다.
 - 인출(fetch)
 - 해독(decode)
 - 실행(execute)
 - 결과 저장 또는 다음 단계 이동
초급 단계에서는 이 네 단계가 "모든 CPU가 완전히 같은 방식으로 동작한다"는 뜻이 아니라, 명령 처리를 이해하기 위한 기본 틀이라는 점을 기억하면 된다.

인터럽트
CPU는 원래 현재 명령 흐름을 따라가지만, 중요한 사건이 생기면 그 흐름을 잠시 끊고 다른 처리를 먼저 해야 한다. 이때 쓰이는 것이 인터럽트다.
대표 예:
 - 타이머 인터럽트
 - 키보드/마우스/장치 인터럽트
 - 저장장치 I/O 완료 알림
 - 네트워크 장치 알림
인터럽트는 현재 흐름을 멈추고 중요한 사건을 먼저 처리하게 하는 메커니즘으로 이해하면 좋다.


좀 더 알아보기: 예외의 종류
예외(exception)는 실행 중 발생하는 내부적 비정상 상황에 더 가깝다.
대표 예:
 - 0으로 나누기
 - 잘못된 메모리 접근
 - 권한 위반
 - 페이지 폴트
초급 단계에서는 다음 정도로 구분해도 충분하다.
 - 인터럽트: 외부나 비동기 사건
 - 예외: 현재 실행 중인 명령과 직접 연결된 내부 사건
조금 더 세밀하게는 다음 용어도 볼 수 있다.
 - fault  - 처리 후 같은 명령을 다시 시도할 수 있는 경우
 - trap  -  처리 후 다음 명령으로 넘어가는 경우
 - abort  - 정상 복구가 어렵고 치명적인 경우
페이지 폴트는 이름만 보면 무조건 장애처럼 들리지만, 실제로는 운영체제가 페이지를 메모리에 올린 뒤 계속 실행할 수 있는 정상적인 예외 처리인 경우가 많다.

확인 문제
프로그램 카운터와 스택 포인터는 왜 역할이 전혀 다른가?
페이지 폴트는 왜 "무조건 장애"가 아니라 예외 처리의 일부가 될 수 있는가?

########5장
5-1 빠른 CPU를 위한 설계 기법
클럭
클럭은 CPU 동작 박자다. 하지만 같은 GHz라고 해서 CPU가 항상 같은 성능을 내는 것은 아니다. 같은 3GHz라도 내부 구조, 캐시, 파이프라인, 명령어 처리 폭, 메모리 대기 시간에 따라 실제 성능은 크게 달라질 수 있다.
즉, 클럭은 중요하지만 클럭만으로 성능을 설명할 수는 없다.

코어와 멀티코어
코어는 명령을 실제로 처리하는 실행 단위다. 멀티코어 CPU는 하나의 패키지 안에 여러 코어를 넣어 병렬성을 높인다.
중요한 점:
 - 코어 수가 많으면 병렬 작업에 유리하다.
 - 모든 프로그램이 코어 수만큼 자동으로 빨라지는 것은 아니다.
 - 직렬 구간이 많은 작업은 코어 수를 늘려도 한계가 있다.

스레드와 멀티스레드
하드웨어 스레드나 SMT는 운영체제가 더 많은 논리 CPU를 보게 한다. 하지만 물리 코어가 두 배가 되는 것은 아니다. 따라서 Thread(s) per core) 수가 커도, 그것을 물리 코어 증가와 같은 의미로 읽으면 안 된다.

IPC와 CPI
좀 더 정확한 성능 설명을 위해 다음 용어를 함께 볼 수 있다.
 - IPC(Instructions Per Cycle) - 한 클럭에 처리하는 명령어 수
 - CPI(Cycles Per Instruction) - 명령 하나당 필요한 평균 클럭 수
같은 클럭이라도 IPC가 높으면 실제 성능이 더 좋을 수 있다. 그래서 클럭 x 구조를 함께 봐야 한다.

5-2장  명령어 병렬 처리 기법 1
명령어 파이프라인
명령어를 한 단계씩 순차적으로 끝내는 대신, 여러 단계를 겹쳐 처리하는 구조가 파이프라인이다. 예를 들어 첫 번째 명령이 실행 단계에 들어갔을 때, 두 번째 명령은 해독 단계, 세 번째 명령은 인출 단계에 들어갈 수 있다.
즉, 파이프라인은 한 명령을 더 빨리 끝내는 기술이라기보다 CPU 자원을 놀리지 않고 겹쳐 쓰는 기술에 가깝다.

파이프라인 해저드
파이프라인은 항상 매끄럽게 흐르지 않는다. 대표적인 방해 요소를 해저드(hazard)라고 부른다.
 - 구조적 해저드  -  같은 자원을 두 명령이 동시에 쓰려는 경우
 - 데이터 해저드  -  앞 명령의 결과가 아직 안 나왔는데 뒤 명령이 그 값을 필요로 하는 경우
 - 제어 해저드  -   분기 명령 때문에 다음에 어느 명령을 가져올지 확실하지 않은 경우

분기 예측
분기 명령이 많으면 파이프라인은 자주 멈춘다. 그래서 CPU는 "아마 다음엔 이쪽으로 갈 것"이라고 예측하며 미리 명령을 가져오기도 한다. 이를 분기 예측(branch prediction)이라고 한다.
예측이 맞으면 성능에 도움이 되지만, 틀리면 이미 진행하던 파이프라인 일부를 버리고 다시 시작해야 한다.

슈퍼스칼라
슈퍼스칼라는 한 클럭에 둘 이상의 명령어를 동시에 발행하려는 구조다. 즉, 파이프라인이 단계 겹치기라면, 슈퍼스칼라는 같은 시점에 더 많이 처리하기에 가깝다.

비순차적 명령어 처리
프로그램 순서대로만 처리하면 자원이 놀 수 있다. 그래서 의존성이 없는 명령은 순서를 조금 바꿔 먼저 처리하는 구조를 쓴다. 이것이 비순차적 명령어 처리(out-of-order execution)다.
초급 단계에서는 다음 정도만 구분하면 충분하다.
 - 파이프라인: 단계 겹치기
 - 슈퍼스칼라: 같은 시점에 더 많이 발행하기
 - 비순차적 처리: 준비된 명령 먼저 처리하기

3가지 키워드로 정리하는 핵심 포인트
  -  명령어 파이프라인
  -  슈퍼스칼라
  -  비순차적 명령어 처리

CISC 와 RISC
CISC
다양하고 강력한 명령어를 활용
상대적으로 적은 수의 명령어로도 프로그램을 실행할수 있다
명령어 파이프라이닝이 불리하다

RISC
명령어의 종류가 적고 짧고 규격화된 명령어 사용
단순하고 적은 수의 고정길이 명령어 집합을 활용 합니다


       CISC                                   RISC
